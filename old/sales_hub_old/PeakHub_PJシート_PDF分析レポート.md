## 1. SWW オンボーディング（p.1〜p.4）

### 会社概要
- **株式会社 START WITH WHY（SWW）**
- 営業文脈で人的リソースを提供するグロース支援スタジオ
- 公式サイト: <https://sw2.jp>
- ミッション: 「アポ獲得」だけでなく、**親和性の高いマーケット発見** と **受注貢献** までを本質支援する

### 役割定義
- **マネージャー**: プロジェクト全体統括・戦略立案・意思決定
- **ディレクター**: 進行管理・クライアント定例・品質担保・育成
- **プレイヤー**: 架電 / メール送付 / アポ取得 / リスト管理 / 報連相

### 使用ツール
- Google Workspace（Drive / Spreadsheet）
- Slack（ディレクター連携・プレイヤー連携）
- ZoomPhone（架電 / 受電）

### 運用ルール（抜粋）
- 日次: 業務終了時に進捗報告
- 週次: チーム定例会
- 緊急時はマネージャーへ直接 Slack ハドル
- メール 24h 以内、Slack は営業時間内即レス
- 守秘義務 / 個人情報の扱いに最大限注意
- 悪い情報ほど即報告（**MUST**）

### 稼働開始 / 終了報告フォーマット
```
@マネージャー @ディレクター
おはようございます！
本日の作業開始いたします。
9:00-10:00 13:00-18:00
```
- 19 時までに日報を提出すること

### 2.3 商材・サービス理解

#### 何を・誰に・いくらで売るか
- **商材**: RPO（戦略設計 / 運用代行 / 日程調整 / 面接代行まで対応）

- **キーワード**: まだ確定していない
- **差別化施策**: お試しのフロント商材

#### 既存顧客の傾向
- 「人事はいるがノウハウがない」会社で成功
- **仮説**: まずは「コスパ良い母集団形成」を訴求する切り口が有効

### 7.1 アカウントプラン

### 7.2 アプローチフェーズ

| --- | --- |

| 次ステップ合意 | 気になる点をピックアップしてヒアリングへ橋渡し |

### 7.3 ファクトファインディング

設計原則: **「問題特定」→「課題設定」** の順／**現状 → 理想** の順でヒアリング。

3 つの刺しポイント:
- 業務効率化
- 売上向上
- コスト削減

> （商材によっては 1 つでも可）

#### 質問例
- 「私たちの ○○ サービスが合いそうだなと感じたのは、主にどの部分の業務がひっ迫しており、効率化したいと考えているのですか？」
- よくある問題① / ② / ③ × その原因① / ② / ③ で課題を構造化

#### 示唆・解決質問
- AI 基盤を他業務に横展開する世界観の提案で **プラスの Yes** を取りに行く

#### テストクロージング
- **弱め**: 「導入したい気持ちが 100 だとしたら、今は何点ですか？」
- **強め**: 「ご懸念がなければ、お申し込みのステップに進めたく、いかがですか？」

#### FAQ 切り返しテンプレ
| よくある反論 | 切り返し |
| --- | --- |
| 忙しい | 「もし時間があり検討できる状態だったとしたら、提案に価値を感じますか？」 |
| 高い | 「相場に対して高いのか／受けられる価値が見えていないのか、どちらに近いですか？」 |
| 検討します | 「上長に共有して首を縦に振りそうですか？○○様視点でどこで躓きそうですか？」 |

### 7.4 オーダーコントロール（**BANT-CHT 拡張**）

> 「いつ何を解決すれば導入できるか？」を **「はぐらかしの仮面（枕詞）」付き** で確認する。

| 軸 | 質問例 |
| --- | --- |
| **B**（予算） | 「100 万円と 500 万円、どちらがご希望に近いですか？」 |
| **A**（決裁者） | 「○○ 円規模なら部長／役員稟議のイメージですか？」「稟議でどなたで躓きそうですか？」 |
| **H**（運用者） | 「導入後、○○様が対面に立つイメージですか？使いこなせそうですか？」 |
| **T**（導入時期） | 「上申しやすい時期で言うといつ頃が会社的に都合よいですか？」 |
| **C**（競合） | 「最終的に導入する決め手はどんなものになりそうですか？」 |

#### ラップアップ（弊社が宿題を抱える形で次回を確定）

## 9. 注意事項・所感

- **エビデンス不足項目**: 「よくある反論・断られる理由（失注理由）」「受注理由」「商談録画・営業資料・送付資料の有無」「リスト作成の要件」「アプローチ NG 企業」「PW 一覧」「キックオフメモ」 など、PDF 上では空欄のままの項目が散見される。トライアル開始後にディレクター主導で埋める必要あり。
- **架電リストの設計**: 1 社につき 10 回まで履歴を横持ちする構造で、コール～コンタクト～資料送付～アポ獲得の各ファネルが集計フラグで自動化されている。Looker / GAS 連携を見据えた設計。
- **戦略仮説**: PJ シート上は「IT・SaaS の方が筋が良い」という結論が示唆されているが、初期実績では製造業 A/B テスト（KAIKA 系・数百名規模）から着手している。**仮説検証ログ** を残し、リスト切替の意思決定速度を担保することが重要。
- **トーク設計**: 商談フレームワーク（p.146〜147）は SWW 全社の標準スクリプトと推察され、PeakHub 案件以外でも横展開可能。**FAQ 切り返し** と **オーダーコントロール（BANT + CHT）** はプレイヤー教育で必読。
