# DB（正本）・Google Drive（提出物）・ローカル（同期/キャッシュ）指針

このドキュメントは、`salse_consulting` ワークスペースで **`nextcrm-app/` と `salesHUB/` と Drive 3 本 + ローカル `_analytics/`** を並行運用する際に、
「どれが正本（Single source of truth）か」「同期はどこが責務を持つか」「スケール（IS人数増）に耐えるための設計」を固定するための指針です。

差分の意図: “同じものが複数にある/どっちを直すべきか分からない” を仕様レベルで潰す。

全体構成（Drive 3 本 + ローカル `_analytics/`）の理由づけは [re-design-2026-05.md](./re-design-2026-05.md) を参照。

## 用語（短く）

- **正本（Single source of truth）**: “最終的に正しい” と定義する唯一の保存先。二重に正本を作らない。
- **ミラー**: 正本を別の場所にコピーして“見える化/オフライン/高速化”するもの。壊れても正本で復旧できる。
- **キュー**: 更新要求を順番待ちさせる仕組み。混雑時に落ちないようにする。
- **冪等性（idempotency）**: 同じ処理を2回実行しても結果が同じ性質。再試行（リトライ）前提で必須。

## 0. 既存ルール（必読）

- **資産の二重保管回避**: `docs/storage-coexistence.md`
- **Driveフル同期（ローカルミラー）**: `scripts/README_drive_sync.md`

この指針は上記の上に “DBとSheets連携” を乗せる目的です（重複を避ける）。

## 1. 正本の割り当て（ここが最優先）

結論（三層）:

- **業務データ（架電ログ・リード・ステータス等）の正本は DB**
- **クライアント提出物（PDF/PPTX/書式付きスプシ）と RAW / 作業物の正本は Drive 3 本（A/B/C）**
- **ローカル `_analytics/` は派生物（解析倉庫）。壊れても再生成できる**

### 1.1 DB（正本）

- **正本**: `salesHUB` / `nextcrm-app` が使う Postgres（用途に応じて別DBでも可）
- **役割**: 更新・集計・権限・監査（ログ）・整合性はDBで担保する
- **禁止**: “スプシを正本にして、DBは従” の二重正本（人数増で破綻しやすい）

### 1.2 Google Drive（提出物/閲覧物の正本）

- **正本**: Drive上のスプシ（見た目・書式・関数）や PDF/PPTX など
- **役割**: クライアントが見たい「形式」を維持する層（提出・閲覧・フォーマット）
- **注意**: Driveの“Googleネイティブ”はローカルに1ファイルとしてコピーできないため、ローカルは「エクスポート」または「ショートカット」で扱う

特に、クライアントと共有する **PJテンプレのスプレッドシート**（例: PeakHUB の PJシート）は、この層（Drive）に置いて「見た目の正本」とする。
アプリ（DB）側はテンプレの **一意キー列**（例: `lead_id` / `clientRowId`）を前提に、差分をバッチ反映して “形を壊さずに中身を更新” する。

### 1.3 ローカル（ミラー / 派生）

ローカルは大きく **2 層**に分かれる。

- **Drive ミラー層（入口名 = ローカルの実ディレクトリ。Git 対象外）**
  - **`project_hangarr/`** → Drive **A**（人 + AI の作業ハブ。CSV export、AI 生成 MD、HTML/CSS 下書き）
  - **`PJ_asset_Data/`** → Drive **B**（重い実体。PDF / PPTX / 録音 / 動画）
  - **`call_rec/`** → Drive **C**（架電 RAW）
- **派生層（再生成可能。Drive へは載せない）**
  - **`_analytics/`**: PJ 横串の KPI（`metrics.json` / `cross_pj_kpi.csv`）と AI 解析（`insights/*.md`）。スキーマは [_analytics-spec.md](./_analytics-spec.md)、音声経路は [voice-pipeline.md](./voice-pipeline.md)。

### 1.4 PJ フォルダ運用（`materials/` と `meta/`）

PJ 配下（例: `PJ_asset_Data/<year>/<pj_slug>/...`、`project_hangarr/<year>/<pj_slug>/...`）は、「人間が迷わない」ことを優先して **資料と説明を同じ場所に寄せる**（命名は `PJname_yyyymmdd` + 年シャード）。

- **`materials/`（B 配下）**: クライアント共有の実ファイル置き場（PDF/PPTX/XLSX 等）と “正本への入口”
  - Googleネイティブ（Sheets/Docs/Slides）でローカルに実体が無い/不安定な場合は、最低限 **`.url` を置いて原本を固定**する
  - 変更や更新が入ったら、Cursor で手動で **MD の更新ログ**を残す（下記）
- **`meta/`（A 配下）**: 人間が読むためのメモ・運用仕様・手順・ADR
  - ここは「ローカルにあるテキスト」を正として読める状態を維持する

#### materials の更新ログ（手動でOK）

運用の最低ラインとして、PJごとに次のどちらかを採用する（どちらでも良いが統一する）。

- **方式A（推奨）**: `materials/00_materials_index.md` に追記（1ファイル集約）
- **方式B**: `materials/notes/YYYY-MM-DD_<topic>.md` を追加（分割）

ログには最低限これを書く:

- 何を追加/差し替えしたか（ファイル名 or Driveリンク）
- いつ・誰が・なぜ（背景1行）
- 影響範囲（クライアント共有/内部のみ）

#### 派生層（`_analytics/`）への波及

PJ 横串の KPI に効く事実（数値・成果・スクリプト変更）は、`_analytics/by-pj/<year>/<pj_slug>/` 配下の `metrics.json` または MD frontmatter にも反映する（スキーマは [_analytics-spec.md](./_analytics-spec.md)）。`master/` は派生物なので、`scripts/build_master_kpi.py` の再実行で更新する。

## 2. Google Sheets 連携の基本方針（200人規模まで見据えた形）

### 2.1 シートは “編集の場” ではなく “表示/提出” に寄せる

200人規模で一番壊れるのは API ではなく運用（上書き競合・誰が正か不明）です。

- **DBが正本**: ユーザーの編集は基本アプリUI → DB
- **シートは出力物**: DBの状態を “準リアルタイム〜定期” で反映
- **例外**: どうしてもシート編集を許す場合は “編集可能列を限定” し、書き戻しは専用の入力面（フォーム）に寄せる（セル直編集は避ける）

### 2.2 更新は「差分集約 + 非同期 + バッチ書き込み」

“人数”より “API呼び出しの粒度と集中” がリスクです。

- **差分集約（coalescing）**: 同じ行（lead等）が短時間に10回更新されても、最後の状態だけを反映する
- **キュー**: PJ単位などで順番待ちさせ、混雑時でも落ちない
- **バッチ**: `spreadsheets.values.batchUpdate` などで 1回の更新にまとめる（1架電=複数セルを1回に）
- **リトライ**: 429（レート制限）は指数バックオフで吸収
- **冪等性**: “同じイベントを2回処理しても同じ結果” にする（再試行前提）

### 2.3 行特定は「行番号」依存にしない

スプシは行挿入/削除が起きるため、**row番号は壊れます**。

- **必須**: シート側に一意キー列（例: `lead_id` / `clientRowId`）を持つ
- **同期側**: 一意キー → 行番号のマップを作ってキャッシュ（TTL数分）し、そこから書く

## 3. 2アプリ（`salesHUB` / `nextcrm-app`）で両立する責務分担

狙いは “Sheets連携の難所（OAuth/レート制限/キュー）を1か所に寄せる” ことです。

### 3.1 推奨: Sheets連携は `salesHUB` 側に集約する

- **`salesHUB`**: Google OAuthトークン管理、Sheets read/write、キュー、バッチ更新、監視
- **`nextcrm-app`**: “連携を呼ぶだけ”（必要なら内部API経由）

理由:

- スコープ/refresh_token/429対策/バッチ等を二重に実装しない
- PJが増えても連携の制御点が1つで済む

### 3.2 例外: `nextcrm-app` 側で独立実装する場合の条件

独立実装は可能ですが、次を満たす必要があります（満たせないなら集約推奨）。

- OAuthスコープ設計（read/write）
- refresh_tokenの永続化と更新
- キュー/リトライ/冪等性/差分集約の実装
- 一意キー列運用の徹底

## 4. “フォルダにリンクを置く” の標準

Driveの正本URLはローカルファイルではないため、参照の入口を固定します。

- **標準**: PJフォルダ直下に `.url`（ショートカット）を置く
- **命名**: `PeakHub_PJシート.url` のように “何の正本か” が一目で分かる名前
- **内容**:

```text
[InternetShortcut]
URL=<Google Sheets / Drive file URL>
```

注意:

- `.url` は “入口” であり “データ” ではない（内容分析にはCSV/XLSXのスナップショットが必要）
- スナップショットは `project_hangarr/<year>/<pj_slug>/pj_sheet/` に日付付きで置く

## 5. スケール指針（50人→200人に伸ばすとき）

### 5.1 SLA（即時性）の分離

- **Fast lane（30秒〜2分）**: 現場が見る最小項目（最新ステータス、最新架電日、次アクションなど）
- **Slow lane（5〜15分/日次）**: 集計列、重い加工、週次レポート

### 5.2 監査と復旧

- **DBに監査ログ**（誰がいつ何を変えたか）を持つ
- Sheetsは壊れても DB から再生成できる前提（“復旧手順”を持つ）

## 6. 仕様決めのテンプレ（このスレで固める用）

新しく「データ/連携/保管」の仕様を決めるときは、最低限これを埋める。

### 6.0 追記ルール（運用）

- **この指針（共通）**: `docs/db-drive-policy.md` を更新する
- **PJ固有の決定**: `project_hangarr/<year>/<pj_slug>/meta/` 配下に ADR を残す（Slack/口頭で決めて終わらない）
- **実装の責務が動く決定**（例: Sheets連携を `salesHUB` に集約する/しない）: ADRに加えて、この指針の「3. 責務分担」にも追記する

### 6.1 決めるべき5点

1. **正本**: DB / Drive / その他（必ず1つ）
2. **更新の入口**: どのUI/どのAPIから更新するか
3. **同期方向**: DB→Sheets / Sheets→DB / 双方向（原則は DB→Sheets）
4. **即時性SLA**: 1分/5分/15分/日次 など
5. **識別子**: 行キー（`lead_id`等）は何か、どこに置くか

### 6.2 ADR（意思決定ログ）ひな形

`project_hangarr/<year>/<pj_slug>/meta/` などに置く想定。

- Title: `ADR-YYYYMMDD-<topic>.md`
- Context: 何が問題か
- Decision: 何を決めたか（正本/同期方向/責務）
- Consequences: 代償・運用上の注意
